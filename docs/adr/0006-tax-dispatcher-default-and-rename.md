# ADR-0006: `package tax` — переименование в `tax_dispatch.rego` и защитный default на случай отсутствия `tax_mode`

## Статус
Принято

## Контекст
Файл `opa/tax.rego` (`package tax`) — единая точка входа,
маршрутизирующая по `input.user.tax_mode`. Два структурных пробела:

1. **Имя файла** не отражало его роль как диспетчера (ожидалось
   `tax_dispatch.rego`) — несущественно функционально, но мешает
   аудитору быстро найти точку входа по имени файла.
2. **Отсутствие `tax_mode`**: если `input.user.tax_mode` отсутствует
   целиком (а не просто содержит невалидное значение), ни одна из двух
   веток маршрутизации (`== "OSNO"` / `!= "OSNO"`) не резолвится —
   `input.user.tax_mode != "OSNO"` с отсутствующим полем сама неопределена,
   а не `true`. В результате `data.tax.decision` возвращает `{}`
   (undefined) вместо безопасного `needs_review: true`. Эмпирически
   подтверждено через `opa eval`: при present-but-invalid значении
   (например, `"FOO"`) маршрутизация в `tax.policy` работает правильно —
   ломается только полностью отсутствующий `tax_mode`.

До сих пор это прикрывалось исключительно внешним JSON Schema гейтом
(`tax_policy_input_validator.dart`), который требует `user.tax_mode`. Но
анти-hallucination инвариант («неопределённость даёт `needs_review:
true`, никогда не падает в `undefined`») должен быть самодостаточным
внутри самого OPA-слоя, а не зависеть целиком от внешнего вызывающего
кода.

## Решение
- Файл переименован: `tax.rego` → `tax_dispatch.rego` (без изменения
  `package tax`).
- Добавлен `default decision := {"needs_review": true, "reason": ...,
  "legal_basis": []}` на уровне диспетчера. Он безопасен: срабатывает
  ТОЛЬКО когда ни одна из двух явных веток маршрутизации не сработала —
  то есть никогда не переопределяет реально посчитанное
  `data.tax.osno.decision` или `data.tax.policy.result`.

## Отклонённые альтернативы
- **Полагаться только на внешнюю JSON Schema валидацию.** Отклонено:
  делает анти-hallucination инвариант не самодостаточным внутри OPA —
  при прямом вызове OPA (например, из `opa eval`, другого сервиса или
  будущей интеграции, минующей `TaxPolicyInputValidator`) система отдала
  бы `undefined` вместо безопасного результата.

## Последствия
- (+) `data.tax.decision` теперь определён при любом входе с валидной
  структурой `user`/`transaction`, даже если `tax_mode` отсутствует.
- (+) Имя файла соответствует его роли — точка входа находится по
  ожидаемому имени.
- (-) Нет: default не покрывает случай отсутствия `input.user` целиком
  (тогда `input.user.tax_mode` обращение к несуществующему полю —
  поведение идентично отсутствию `tax_mode`, default сработает и здесь,
  так что регрессии нет).
