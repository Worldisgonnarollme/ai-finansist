# ADR-0001: ОСНО — отдельное дерево policy-пакетов с границей «квалификация, не расчёт»

## Статус
Принято

## Контекст
НПД/УСН/АУСН живут в одном файле `opa/tax_policy.rego` (`package tax.policy`):
одно плоское правило `result` на каждую комбинацию режима и направления
операции, в выводе которого уже есть `tax_rate` (ставка известна заранее и
фиксирована законом — 4/6/8/15%).

ОСНО устроено иначе: ставка НДФЛ зависит от накопленного дохода (13→22%),
ставка НДС — от вида операции (0/10/20%), а право на вычет расхода — от
документа. Это не одно решение, а три разных юридических вопроса (объект
налога на прибыль/НДФЛ, объект НДС, вычитаемость расхода), которые при этом
должны сходиться в одно агрегированное решение по операции.

## Решение
ОСНО получает собственное дерево OPA-пакетов:
`tax.osno.income_tax`, `tax.osno.vat`, `tax.osno.expenses` — каждый отвечает
ровно на один юридический вопрос и возвращает только булеву квалификацию
(`is_taxable_income` / `vat_object` / `deductible`) + `needs_review` +
`legal_basis`. `tax.osno` агрегирует их в `decision`, ничего не пересчитывая.

Сумма налога и процентная ставка не вычисляются и не угадываются ни в одном
из этих пакетов — это зафиксировано CI-гардом
(`.github/workflows/opa-osno-guard.yml`), а не только комментарием.

Маршрутизация по `input.user.tax_mode` идёт через единую точку входа
`package tax` (`opa/tax.rego`): ОСНО — в `data.tax.osno.decision`, всё
остальное — в `data.tax.policy.result`. У диспетчера нет своего default:
`needs_review`, рассчитанный внутри ОСНО, доходит до верха без изменений.

## Отклонённые альтернативы
- **Добавить правила ОСНО в `tax.policy` по аналогии с НПД/УСН.** Отклонено:
  потребовало бы либо смешать в одном правиле три независимых юридических
  вопроса (доход/НДС/расход), либо вписать в общий вывод поле `tax_rate` —
  а ставка НДФЛ по ОСНО не константа режима, а функция от накопленного
  дохода, посчитать которую в policy означало бы выйти за рамки
  квалификации в расчёт.
- **Считать сумму налога прямо в policy** (raised at design time). Отклонено
  принципиально: OPA отвечает на вопрос «облагается ли», а не «сколько
  платить» — расчёт суммы это отдельный слой с отдельной ответственностью
  и отдельными тестами.

## Последствия
- (+) Любая будущая попытка вписать в ОСНО-пакеты расчёт или ставку ловится
  CI до мерджа, а не на код-ревью.
- (+) Юридическое основание (`legal_basis`) и решение о готовности
  (`needs_review`) для каждого из трёх вопросов разделены — проще
  показать аудитору, на основании чего принято каждое конкретное решение.
- (-) Известное ограничение агрегатора: `tax.osno.decision.needs_review`
  физически не может стать `false` для одной транзакции, потому что
  `income_tax` требует `direction == "income"`, а `expenses` —
  `direction == "expense"` одновременно. До боевого использования это
  нужно поправить (сделать релевантным только тот под-пакет, который
  соответствует направлению операции).
