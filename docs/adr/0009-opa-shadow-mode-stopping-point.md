# ADR-0009: Точка остановки — OPA-интеграция завершена на этапе passive diagnostics

## Статус
Принято

## Контекст
`OpaClient`/`_runOpaShadow` (см. `lib/services/opa_client.dart`,
`lib/app_state.dart`) реализованы и протестированы: HTTPS-only, валидация
входа до сети, request_id, обработка всех ошибок без утечки в основной
флоу (см. чек-лист проверки от 2026-06-29 в памяти проекта). Это создаёт
соблазн считать интеграцию "почти готовой к enforce" и продолжить
рефакторинг в эту сторону по инерции.

## Решение
Shadow-mode интеграция OPA считается **завершённой** на этапе passive
diagnostics. Зафиксировано явно, чтобы не быть переоткрытым по
умолчанию:

- OPA не участвует в расчётах — ни одно число в `TaxResult`/
  `TaxCalculator` не зависит от `OpaDecision`.
- OPA не участвует в пользовательских решениях — `OpaShadowOutcome`
  читается только логгером (`opa_shadow_logger.dart`), UI его не видит.
- Дальнейшие шаги (enforce-режим, расширение `Transaction` под
  структурные факты схемы, и т.п.) предпринимаются **только после**
  накопления статистики mismatch/skip в продакшен-логах — не по плану
  "раз код готов, продолжим".

## Отклонённые альтернативы
- **Сразу спроектировать enforce-ready архитектуру "на будущее".**
  Отклонено: до накопления реальной статистики неизвестно, какой именно
  enforce-флоу нужен (блокировать классификацию? только помечать
  needs_review? для каких режимов?) — проектировать это сейчас означало
  бы угадывать требования.

## Последствия
- (+) Явная граница для future-self и других разработчиков: следующий
  шаг в `opa_client.dart`/`app_state.dart` без статистики mismatch/skip
  — это незапрошенное расширение скоупа, а не продолжение текущей задачи.
- (-) Известные нерешённые ограничения (100% SKIP пока `Transaction` не
  хранит related_to_business/linked_document/has_vat_invoice и т.п.;
  enforce требует рефакторинга, не флага) остаются открытыми осознанно,
  не как недосмотр.
