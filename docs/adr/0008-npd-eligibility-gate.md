# ADR-0008: НПД — eligibility-гейт (`tax.npd`) перед существующей классификацией ставок в `tax.policy`

## Статус
Принято

## Контекст
До этого решения НПД квалифицировался напрямую в `tax.policy` (3 правила:
доход от юрлица/платформы → 6%, доход от физлица с документом → 4%, доход
от физлица с текстовым признаком без документа → `needs_review: true`).
Эти правила корректно отвечают на вопрос «по какой ставке облагается
доход», но НИГДЕ не проверяется отдельный вопрос «имеем ли мы право
применять НПД вообще» — ст. 4 ч. 2 422-ФЗ от 27.11.2018 содержит закрытый
перечень условий, исключающих режим (лимит дохода 2,4 млн ₽/год, наёмные
работники, перепродажа, подакцизные/маркированные товары, добыча
полезных ископаемых, агентские схемы без исключения, совмещение с другим
режимом по той же деятельности). Раньше операция могла быть уверенно
классифицирована по ставке 6% даже если плательщик давно превысил лимит
или нанял работников — система не имела структурного способа это
заметить. Это ровно тот же класс пробела, что был закрыт для АУСН/ЕСХН/
ПСН в ADR-0007.

## Решение
Создан `tax.npd` (`tax_npd.rego`) с правилом `eligibility` (модель
`compliant`/deny-default — по аналогии с ADR-0007, а не `needs_review`,
так как нарушение лимита/запрета — это уверенный факт исключения режима,
а не неопределённость) и агрегирующим `decision`, который:

- при `eligibility.compliant == true` возвращает `data.tax.policy.result`
  **без изменений** — классификация ставок 4%/6% остаётся единственно в
  `tax.policy`, не дублируется и не переписывается;
- при `eligibility.compliant == false` блокирует автоматическое
  применение классификации (`needs_review: true` с конкретной причиной
  нарушения) — даже если `tax.policy` сам по себе уверенно определил бы
  ставку, операция требует переквалификации по другому режиму;
- для расходных операций возвращает уверенный `needs_review: false`
  факт «расходы не являются категорией НПД» (ст. 8 422-ФЗ — налоговая
  база НПД — доход целиком, без уменьшения на расходы).

`tax_dispatch.rego` теперь маршрutизирует `NPD` в `tax.npd.decision`, а не
напрямую в `tax.policy.result`. УСН остаётся в `tax.policy` без
изменений — у него такой же пробел (ст. 346.12 НК РФ) потенциально есть,
но это отдельное решение, не покрытое данным ADR.

## Отклонённые альтернативы
- **Добавить проверку лимита прямо в существующие правила `tax.policy`
  для НПД.** Отклонено: смешало бы два разных юридических вопроса (ставка
  по ст. 8 vs право на режим по ст. 4) в одном правиле и нарушило бы уже
  протестированную, рабочую логику классификации без необходимости —
  именно то нерекомендуемое смешение, которого ADR-0001/0007 учат
  избегать.
- **Скопировать/переписать классификацию ставок в `tax.npd`.** Отклонено:
  создало бы два источника истины для одного вопроса (ставка 4%/6%).
  `tax.npd` переиспользует `data.tax.policy.result` напрямую.

## Последствия
- (+) Превышение лимита НПД или наличие работников теперь структурно
  блокирует автоматическое применение ставки 4%/6%, даже если
  классификация контрагента сама по себе была уверенной.
- (+) Классификация ставок осталась нетронутой (ноль регрессий в
  `tax_policy_test.rego`) — eligibility добавлена как гейт сверху, а не
  переписыванием существующей логики.
- (-) `decision` для НПД теперь имеет ДВЕ разные формы в зависимости от
  eligibility: либо ровно форма `tax.policy.result` (`category`/
  `taxable`/`tax_rate`/`needs_review`), либо форма `{eligibility,
  needs_review, reason, legal_basis}` без `tax_rate` — вызывающий код не
  может полагаться на единую форму без проверки `needs_review` первой.
- (-) `tax_policy_input.schema.json` требует `context.npd` при
  `tax_mode == "NPD"` — любой существующий вызывающий код (когда OPA-слой
  будет подключён к приложению) должен будет передавать эти 7 новых
  фактов, которых раньше не было.
