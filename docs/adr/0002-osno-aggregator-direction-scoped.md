# ADR-0002: Агрегатор `tax.osno.decision` — отдельные правила по direction вместо одного на все три под-пакета

## Статус
Принято. Заменяет (не правит) последствие, описанное в ADR-0001.

## Контекст
Исходный агрегатор требовал одновременной уверенности (`needs_review ==
false`) во всех трёх под-пакетах — `income_tax`, `vat`, `expenses` — чтобы
выдать `decision.needs_review == false`. Но `income_tax`/`vat` структурно
требуют `direction == "income"`, а `expenses` — `direction == "expense"`.
Для одной транзакции `direction` не может быть одновременно тем и другим,
поэтому уверенная ветка была физически недостижима, и `decision` всегда
падал на статический default — независимо от того, что реально посчитали
под-пакеты.

Default дополнительно фабриковал поля (`income_tax.is_taxable_income:
false`, `vat.vat_object: false`, `expense.deductible: false`), которые не
отражали то, что под-пакеты реально вычислили — то есть default лгал, а
не честно сигнализировал «недостаточно данных».

## Решение
`decision` определяется отдельными правилами в зависимости от
`input.transaction.direction`:
- `direction == "income"` → требует уверенности `income_tax` и `vat`,
  выдаёт только эти два поля.
- `direction == "expense"` → требует уверенности `expenses`, выдаёт
  только это поле.

Default — это `{"needs_review": true}`, без фабрикации полей под-пакетов,
которые не относятся к фактическому `direction` операции.

## Отклонённые альтернативы
- **Оставить требование уверенности во всех трёх под-пакетах.** Отклонено:
  это было бы постоянным структурным дефектом, ловящим в `needs_review`
  100% операций независимо от качества входных данных.
- **В default подставлять последние реально посчитанные значения
  под-пакетов вместо `false`.** Отклонено: даже частично уверенный
  результат (например, `income_tax` уверен, `vat` нет) не означает, что
  decision готов — выдавать половину диагноза как часть «решения» создало
  бы риск, что вызывающий код примет частичный результат за полный.

## Последствия
- (+) `decision.needs_review == false` теперь физически достижим для
  реальных операций — устранён дефект, описанный как известное
  ограничение в ADR-0001.
- (+) Default больше не выдаёт чужой расчёт под-пакета за `false` —
  отсутствие поля честно сигнализирует «не относится/не считалось»,
  а не «проверено и не облагается».
- (-) Форма `decision` теперь зависит от `direction`: вызывающий код не
  может полагаться на присутствие `income_tax`/`vat`/`expense`
  одновременно — нужно сначала проверить `needs_review`, затем взять
  только релевантные для данного `direction` поля.
